# POV Runner Constructor — передача проекта на другое устройство

Дата состояния: 21 июля 2026 года.

## 1. Назначение проекта

POV Runner Constructor — проект на Godot 4.7 для создания POV-runner роликов.
Камера автоматически движется вперёд по трём полосам. Игрок выполняет JUMP,
LEFT, RIGHT, DUCK и SMASH. Конструктор позволяет выбирать длительность,
наполнение раунда, модели, окружение и масштаб объектов.

Репозиторий:

`https://github.com/mikhailliachenko-hue/POVRunnerConstructor`

Основная сцена: `main.tscn`.

Основная логика: `scripts/main.gd`.

## 2. Технологический стек

- Godot Engine 4.7 stable;
- GDScript;
- Forward+/Compatibility renderer в зависимости от запуска;
- GLB/GLTF через `GLTFDocument` и `GLTFState`;
- `ArrayMesh` для производной фильтрации треугольников;
- `ConfigFile` для настроек раундов, моделей и импортированных миров;
- стандартные Godot `MeshInstance3D`, `Camera3D`, `Environment`, UI Control;
- Git и GitHub;
- Blender/Python могут использоваться для будущей офлайн-обработки GLB, но
  сейчас не являются runtime-зависимостями проекта;
- сторонних библиотек и сетевых runtime-сервисов нет.

Использовавшийся Godot на исходном компьютере:

`D:\Godot_v4.7-stable_win64.exe\Godot_v4.7-stable_win64_console.exe`

На другом устройстве путь может отличаться. Нужна именно Godot 4.7 stable.

## 3. Что уже работает

### Основной runner

- автоматическое движение вперёд;
- три полосы `X = -3.2 / 0 / 3.2`;
- JUMP, LEFT, RIGHT, DUCK, SMASH;
- подсказки, иконки, обратный отсчёт, прогресс, финиш и restart;
- препятствия и отдельные GLB/GLTF-модели раундов;
- визуальный спутник-бегун;
- загрузка звуков, текстур и пользовательских моделей;
- конструктор длительности и наполнения раунда.

### Imported World Mode

Ключевые файлы:

- `scripts/imported_world_manager.gd`;
- `scripts/world_import_smoke_test.gd`;
- `scripts/round_02_visual_test.gd`;
- `миры/the_end_world/world.glb`;
- `миры/the_end_world/world_config.cfg`;
- `миры/README_RU.md`.

Система:

- автоматически находит папки миров в `миры/<world_id>`;
- находит первый GLB/GLTF либо использует `model_file` из конфигурации;
- загружает выбранный мир один раз при построении раунда;
- освобождает предыдущий экземпляр при перестроении;
- рассчитывает world-space AABB всех видимых MeshInstance3D;
- считает меши и вершины;
- извлекает title/author/license/source из GLB asset extras;
- выбирает продольную ось X/Z;
- применяет только равномерный масштаб;
- сохраняет настройки мира;
- рассчитывает характерную высоту поверхности по percentile вершин, а не по
  нижнему или верхнему экстремуму AABB;
- привязан к `round_02`;
- отключает стандартное процедурное окружение для импортированного мира;
- использует приглушённую дорогу вместо розовой Candy-дороги;
- создаёт диагностические игровые кадры и contact sheet.

### Текущий тестовый мир

- название: Minecraft world;
- автор: guus — `https://sketchfab.com/guusvanhouten1`;
- источник: `https://sketchfab.com/3d-models/minecraft-world-8ee90583b45749c387a9c45e95031cd1`;
- лицензия: CC BY 4.0;
- 63 меша;
- 903 457 вершин;
- исходный Godot AABB приблизительно `7.823 × 0.951 × 5.428`;
- продольная ось: X, развёрнутая вдоль трассы;
- масштаб для 720 м: приблизительно `94.59263`;
- итоговый AABB приблизительно `513.449 × 89.925 × 740.000 м`.

Исходный GLB нельзя перезаписывать или удалять.

## 4. Последняя реализованная функция: расчистка трассы

В `ImportedWorldManager.carve_route_corridor()` реализована одноразовая
фильтрация треугольников импортированного мира.

Текущие настройки:

- `carve_route=true`;
- `carve_width=9.2`;
- `carve_height=6.2`;
- коридор проходит по всей длине трассы;
- проверяется пересечение triangle AABB с объёмом маршрута;
- пересекающиеся индексы не включаются в производный `ArrayMesh`;
- материалы поверхностей переносятся в новый mesh;
- исходный GLB остаётся неизменным;
- боковая геометрия за пределами коридора сохраняется;
- обработка выполняется только при построении, не каждый кадр.

В последнем автоматическом визуальном тесте было исключено 390 треугольников.

## 5. Главная нерешённая проблема

Пользователь сообщает, что в реальном запуске внутри дорожки всё ещё остаются
отдельные текстурированные элементы/«текстурки». На скриншотах это прежде всего
высокая трава и фрагменты блочного склона, вторгающиеся в левую часть маршрута.

Последний автоматический кадр `route_00_start.png` после фильтрации выглядел
чисто, но пользователь всё равно видит остатки. Следовательно, текущая проверка
по triangle AABB недостаточна либо реальный кадр отличается по позиции,
параметрам/моменту от автоматического теста.

Вероятные причины:

1. Некоторые поверхности имеют неиндексированную, strip/fan или нестандартную
   геометрию и пропускаются текущей обработкой.
2. Остаток является alpha-текстурой травы на quad, чьи вершины находятся за
   границей, а прозрачная часть/плоскость визуально входит в коридор.
3. Проверка AABB треугольника слишком грубая по одним объектам и недостаточно
   точная по другим.
4. Реальная левая/правая граница камеры шире `carve_width=9.2` из-за FOV,
   уклонения, препятствия или перспективы.
5. В момент кадра мир перестроен со старыми настройками либо загружена старая
   версия проекта/кэша.
6. Часть видимого объекта может быть отдельным billboard, MultiMesh,
   GPUParticles либо типом узла, отличным от MeshInstance3D.
7. Визуальная трава может принадлежать материалу с alpha, где геометрическая
   плоскость пересекает границу коридора, хотя непрозрачные пиксели оказываются
   внутри него.

## 6. Рекомендуемое следующее решение

Не удалять исходный GLB вручную. Сначала точно определить источник остатка.

### Этап A — воспроизводимая диагностика

1. Запустить `round_02` с теми же настройками, что у пользователя.
2. На проблемном кадре временно вывести:
   - имя MeshInstance3D;
   - имя mesh/material;
   - world-space AABB;
   - число поверхности и треугольника под центральным/левым raycast;
   - границы carve-коридора.
3. Добавить диагностический полупрозрачный BoxMesh для carve-объёма.
4. Сделать кадры центральной, левой и правой полосы на дистанциях 0, 15, 30,
   60 м, включая JUMP и DUCK.
5. Убедиться, что `carved=<число>` появляется в консоли реального запуска.

### Этап B — надёжное вырезание

Предпочтительный вариант: создать производный очищенный GLB офлайн, а не
пересобирать 903 тысячи вершин при каждом запуске.

1. Импортировать копию `world.glb` в Blender через Python.
2. После применения того же uniform transform создать прямоугольный объём
   safe corridor.
3. Для обычной непрозрачной геометрии выполнить Boolean Difference или
   bmesh-фильтрацию реальных полигонов.
4. Для alpha-card травы:
   - отделить vegetation material slots;
   - удалить целые quads, пересекающие коридор с небольшим запасом;
   - не пытаться резать прозрачную текстуру по пикселям в runtime.
5. Закрыть края рельефа вертикальными блоками/подпорными поверхностями, чтобы
   не было открытых дыр и backfaces.
6. Экспортировать `world_route_clean.glb` рядом с исходником.
7. Изменить только `model_file` в `world_config.cfg` на производную копию.
8. Сохранить лицензионные extras/attribution.
9. Исходный `world.glb` оставить без изменений.

Альтернативный runtime-вариант — точное triangle-box clipping с разрезанием
треугольников на границе коридора, но это сложнее, медленнее и хуже кэшируется.
Его использовать только если офлайн-производная сцена неприемлема.

### Этап C — кэш анализа

После стабильного вырезания сохранить рядом с миром кэш:

- hash/mtime исходного GLB;
- исходный и итоговый AABB;
- выбранную ось и transform;
- surface percentile;
- число удалённых мешей/треугольников;
- путь очищенной производной сцены;
- предупреждения и лицензию.

Повторно анализировать мир только если изменился GLB или настройки коридора.

```powershell
& 'D:\Godot_v4.7-stable_win64.exe\Godot_v4.7-stable_win64_console.exe' --headless --path 'D:\POVRunnerConstructor' --editor --quit
```

```powershell
& 'D:\Godot_v4.7-stable_win64.exe\Godot_v4.7-stable_win64_console.exe' --headless --path 'D:\POVRunnerConstructor' --quit-after 120
```

```powershell
& 'D:\Godot_v4.7-stable_win64.exe\Godot_v4.7-stable_win64_console.exe' --headless --path 'D:\POVRunnerConstructor' --script 'res://scripts/world_import_smoke_test.gd'
```

Для реальных PNG нужен графический renderer, не `--headless`:

```powershell
& 'D:\Godot_v4.7-stable_win64.exe\Godot_v4.7-stable_win64_console.exe' --path 'D:\POVRunnerConstructor' --resolution 1158x633 --script 'res://scripts/round_02_visual_test.gd'
```

Headless dummy renderer может печатать RID/ObjectDB leak warnings при
принудительном завершении. Оценивай отдельно exit code и ошибки GDScript.

## 11. Как продолжить на другом устройстве

1. Установить Git и Godot 4.7 stable.
2. Клонировать репозиторий.
3. Открыть папку проекта в Godot и дождаться полного импорта 35-МБ GLB.
4. Запустить editor-check.
5. Запустить smoke-test.
6. Открыть игру, выбрать `round_02`, убедиться, что IMPORTED WORLD включён.
7. Воспроизвести остаточную «текстурку» и сделать кадр с дистанцией/полосой.
8. Начать с раздела 6 этого документа, не переписывая всю систему заново.

## 12. Git и осторожность

- Перед работой всегда выполнять `git status --short`.
- Не удалять и не перезаписывать исходный GLB.
- Не удалять пользовательские изменения без явного разрешения.
- Не коммитить `.godot`, локальные логи, shader cache и записанные AVI.
- Не устанавливать зависимости без разрешения.
- Не менять механику действий одновременно с Imported World.
- После каждого изменения запускать editor-check и runtime/smoke-test.
